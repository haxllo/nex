## Existing Tray Implementation (Reference)

The legacy tray implementation lives in `apps/core/src/windows_overlay/tray.rs` (273 lines). Key design elements to preserve:

1. **Message-only window pattern**: Creates a `HWND_MESSAGE` window (`NexTrayMessageWindow` class) that receives `NEX_WM_TRAY_ICON` callbacks from `Shell_NotifyIconW`. The window stores state via `GWLP_USERDATA`.

2. **Icon loading**: Uses `ExtractIconExW` to pull the small icon (index 0) from the running .exe.

3. **Tooltip**: Formats based on `game_mode_enabled` and `hotkey_issue_active` state.

4. **Context menu**: Uses `CreatePopupMenu` / `AppendMenuW` with `MF_STRING | MF_CHECKED` for Game Mode toggle. Uses `TrackPopupMenu(TPM_LEFTALIGN | TPM_RIGHTBUTTON | TPM_RETURNCMD)`.

5. **Menu actions map** (from legacy `window.rs` WM_COMMAND handler):
   - TRAY_MENU_SHOW → NEX_WM_EXTERNAL_SHOW → OverlayEvent::ExternalShow
   - TRAY_MENU_OPEN_CONFIG → `ShellExecuteW("open", config_path)` (direct, no event)
   - TRAY_MENU_CHECK_UPDATES → OverlayEvent::TrayCheckForUpdates
   - TRAY_MENU_GAME_MODE → OverlayEvent::TrayToggleGameMode
   - TRAY_MENU_QUIT → OverlayEvent::ExternalQuit

6. **Left-click behavior**: `WM_LBUTTONUP` / `WM_LBUTTONDBLCLK` → OverlayEvent::ExternalShow

## Critical Difference from Legacy

The legacy tray posts `WM_COMMAND` messages to the overlay window's WndProc, which then posts synthetic `NEX_WM_TRAY_*` messages. The new tray MUST send `OverlayEvent` variants directly through the crossbeam channel instead. The `runtime_loop.rs` worker thread already handles all needed events (`ExternalShow`, `ExternalQuit`, `TrayToggleGameMode`, `TrayCheckForUpdates`) in its `on_event` method.

## Iced Overlay Context

The Iced window is created by `boot::run()` on the main thread. The tray does not need the Iced window's HWND — it uses a standalone `HWND_MESSAGE` window. `TrackPopupMenu` with `TPM_RETURNCMD` returns the selection directly (no WM_COMMAND routing needed), so the parent window is only used for `SetForegroundWindow` to dismiss the menu on outside click. Using the tray message window itself as the foreground window is sufficient.

Create `apps/core/src/overlay/tray.rs` with a self-contained `TrayIcon` struct. This module is pure Win32 (no Iced dependency) and communicates through the `crossbeam_channel::Sender<OverlayEvent>`.

**Structural design:**

```rust
#![cfg(target_os = "windows")]

use std::sync::{Arc, Mutex};
use crossbeam_channel::Sender;
use crate::overlay::model::OverlayEvent;

struct TrayState {
    event_tx: Sender<OverlayEvent>,
    config_path: String,
    game_mode_enabled: bool,
    hotkey_issue_active: bool,
}

pub(crate) struct TrayIcon {
    message_hwnd: HWND,      // HWND_MESSAGE window for Shell_NotifyIcon callbacks
    icon_handle: isize,       // HICON from ExtractIconExW
    icon_added: bool,
    state: Arc<Mutex<TrayState>>,
}
```

**Required methods:**

- `TrayIcon::create(event_tx: Sender<OverlayEvent>, config_path: &str) -> Result<Self, String>`
  - Register `NexTrayMessageWindow` window class with a static WndProc
  - Create `HWND_MESSAGE` window via `CreateWindowExW(0, class_name, NULL, 0, 0, 0, 0, 0, HWND_MESSAGE, NULL, instance, NULL)`
  - Allocate `TrayState` on heap, store pointer in `GWLP_USERDATA`
  - Load icon via `ExtractIconExW(exe_path, 0, &mut large, &mut small, 1)`. Destroy large icon immediately; keep small.
  - Call `Shell_NotifyIconW(NIM_ADD, &data)` with `NIF_MESSAGE | NIF_ICON | NIF_TIP` flags
  - `uCallbackMessage` must be `NEX_WM_TRAY_ICON` (WM_APP + 18)
  - `uID` must be `TRAY_ICON_ID` (1)
  - Tooltip initial text: `"Nex Launcher"`

- `TrayIcon::set_game_mode(&self, enabled: bool)`
  - Update `TrayState::game_mode_enabled`
  - Call `Shell_NotifyIconW(NIM_MODIFY, ...)` with updated tooltip (via `update_tray_tooltip` internal helper)

- `TrayIcon::set_hotkey_issue(&self, active: bool)`
  - Update `TrayState::hotkey_issue_active`
  - Call `Shell_NotifyIconW(NIM_MODIFY, ...)` with updated tooltip

- `TrayIcon::remove(&mut self)` (called by Drop, but explicit for early cleanup)
  - `Shell_NotifyIconW(NIM_DELETE, ...)`
  - `DestroyIcon(icon_handle)`
  - `DestroyWindow(message_hwnd)`

**Static WndProc (unsafe extern "system" fn):**
- Store `TrayState*` on the window via `GWLP_USERDATA` at `WM_CREATE` time. Retrieve via `GetWindowLongPtrW(hwnd, GWLP_USERDATA)` on each message.
- On `WM_DESTROY`: drop the `Box<TrayState>`, set `GWLP_USERDATA` to null, return 0.
- On `NEX_WM_TRAY_ICON` (WM_APP + 18):
  - `lparam == WM_LBUTTONUP || lparam == WM_LBUTTONDBLCLK` → `state.event_tx.send(OverlayEvent::ExternalShow).ok()`
  - `lparam == WM_RBUTTONUP` → call internal `show_context_menu(hwnd, state)` helper
- On `WM_COMMAND` (posted by the context menu helper): handle menu IDs as described below.

**Internal context menu helper (`show_context_menu`):**
- Create popup menu with items: `"Open Nex"` (TRAY_MENU_SHOW=41001), `"Open Config"` (41002), `"Check for Updates"` (41003), separator, `"Game Mode"` (41004, MF_CHECKED if game_mode_enabled), separator, `"Quit"` (41005)
- Call `SetForegroundWindow(message_hwnd)` then `TrackPopupMenu(menu, TPM_LEFTALIGN | TPM_RIGHTBUTTON | TPM_RETURNCMD, cursor.x, cursor.y, 0, message_hwnd, NULL)`
- On non-zero return:
  - 41001 (Show) → `event_tx.send(OverlayEvent::ExternalShow).ok()`
  - 41002 (Open Config) → `ShellExecuteW(NULL, "open", config_path, NULL, NULL, SW_SHOW=5)` (same pattern as legacy `open_help_config_file`)
  - 41003 (Check Updates) → `event_tx.send(OverlayEvent::TrayCheckForUpdates).ok()`
  - 41004 (Game Mode) → `event_tx.send(OverlayEvent::TrayToggleGameMode).ok()`
  - 41005 (Quit) → `event_tx.send(OverlayEvent::ExternalQuit).ok()`
- Destroy menu after

**Tooltip format (match legacy `tray_tooltip_text`):**
```rust
fn tooltip_text(game_mode: bool, hotkey_issue: bool) -> String {
    if hotkey_issue { "Nex (hotkey issue)" }
    else if game_mode { "Nex (Game Mode)" }
    else { "Nex Launcher" }
}.to_string()
```

**Drop impl:** Clean up tray icon and window. Do NOT panic if already removed — guard with `icon_added` flag.

**windows-sys imports needed** (all already in Cargo.toml features):
- `Win32_UI_Shell`: `Shell_NotifyIconW`, `NOTIFYICONDATAW`, `NIF_*`, `NIM_*`, `ExtractIconExW`, `ShellExecuteW`
- `Win32_UI_WindowsAndMessaging`: `CreateWindowExW`, `RegisterClassW`, `WNDCLASSW`, `DefWindowProcW`, `DestroyWindow`, `GetCursorPos`, `CreatePopupMenu`, `AppendMenuW`, `TrackPopupMenu`, `DestroyMenu`, `SetForegroundWindow`, `PostMessageW`, `DestroyIcon`, `GetModuleHandleW`, `SetWindowLongPtrW`, `GetWindowLongPtrW`, `HWND_MESSAGE`, `WM_*`, `MF_*`, `TPM_*`
- `Win32_Foundation`: `HWND`, `LPARAM`, `LRESULT`, `WPARAM`, `GetLastError`, `POINT`
- Required constants: `WM_APP = 0x8000`, `NEX_WM_TRAY_ICON = WM_APP + 18`, `TRAY_ICON_ID = 1`

**No test module** — the tray is inherently visual/OS-integrated and uses raw Win32 FFI. Testing is manual via smoke test.

**Pattern to follow:** Use `let mut wide: Vec<u16> = s.encode_utf16().chain(std::iter::once(0)).collect();` for string-to-wide conversion (consistent with existing `overlay/platform.rs::to_wide`).

```rust

std::thread::Builder::new()
    .name("nex-tray-updater".into())
    .spawn(move || {
        loop {
            crossbeam_channel::select! {
                recv(tray_gm_rx) -> msg => {
                    if let Ok(enabled) = msg {
                        tray_icon.set_game_mode(enabled);

                recv(tray_hi_rx) -> msg => {

                    } else { break; }
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tray → Event Channel | An attacker with code execution in the process could forge OverlayEvent variants through the unbounded channel, but this is within the same process — no privilege escalation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | TrayIcon context menu thread | accept | The tray icon runs in-process. Tampering requires code execution. The tray updates game mode by sending OverlayEvent::TrayToggleGameMode — the runtime worker validates this identically to keyboard/menu triggers. |
| T-01-02 | Information Disclosure | Tray tooltip text | accept | Tooltip shows "Nex (Game Mode)" or "Nex (hotkey issue)". This is intentionally visible UI state, not sensitive information. |
| T-01-03 | Denial of Service | Shell_NotifyIconW calls | accept | Rapid Shell_NotifyIconW calls could theoretically cause explorer.exe to throttle, but the tray only updates on state transitions (game mode toggle, hotkey issue change) — at most a few calls per session. |
</threat_model>
