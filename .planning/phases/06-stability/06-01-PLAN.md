<objective>
Make the application resilient to three crash vectors: Tantivy panics during search, corrupt icon file decoding, and unexpected hotkey thread termination. Convert hard crashes into graceful degradation with logged errors and automatic recovery where possible.

Purpose: Prevent the application from crashing to desktop when internal subsystems encounter unexpected errors. A crashed search worker should return error results, not kill the process. A dead hotkey thread should restart, not silently stop responding to hotkeys.
Output: Modified search_worker.rs, icons.rs, hotkey.rs, and runtime_loop.rs with panic guards, health checks, and restart logic.
</objective>

Wrap the `image::load_from_memory(&bytes)` call in `decode()` with `std::panic::catch_unwind`. The `image` crate can panic on certain malformed image data that passes basic file reads but triggers internal asserts.

**Specific changes:**
1. In the `decode` function (lines 125-135), replace:
```rust
let img = image::load_from_memory(&bytes).ok()?;
```
with a catch_unwind guard:
```rust
let img = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
    image::load_from_memory(&bytes)
}))
.ok()
.and_then(|result| result.ok())?;
```
This collapses: panic caught → None, load error → None, success → Some(img). The existing `?` propagates None to return None from `decode`, which causes `resolve()` to return None (icon not found → placeholder rendered).

2. Add `use std::panic::AssertUnwindSafe;` if not present.

3. **Do NOT** change any other logic in `resolve()`, `get_image()`, or `prefetch_rows()`. The existing fallback chain (None → empty placeholder in view.rs `result_icon`) already handles decode failures.
</action>

```rust
pub(crate) fn is_alive(&self) -> bool {
    self.inner.as_ref()

        .map(|h| !h.is_finished())
```

```rust

```
When the HotkeyListener is successfully created, store it:


4. Add a `hotkey_listener: Arc<Mutex<Option<HotkeyListener>>>` field to `RuntimeWorker`.

5. In `RuntimeWorker::on_event`, before the `match event` block, add a periodic health check (every ~32nd invocation, using a counter to avoid checking every event):
```rust
// Check hotkey thread health every ~32 events
{
    static CHECK_COUNTER: std::sync::atomic::AtomicU32 = std::sync::atomic::AtomicU32::new(0);
    let count = CHECK_COUNTER.fetch_add(1, std::sync::atomic::Ordering::Relaxed);
    if count % 32 == 0 {
        if let Ok(guard) = self.hotkey_listener.lock() {
            if let Some(ref listener) = *guard {
                if !listener.is_alive() {
                    log_warn("[nex] hotkey listener thread died — attempting restart");
                    // Drop the dead listener
                    drop(guard);
                    if let Ok(mut guard) = self.hotkey_listener.lock() {
                        *guard = None;
                    }
                    // Attempt restart
                    match HotkeyListener::start(&self.runtime_config.hotkey, /* event_tx */) {
                        Ok(new_listener) => {
                            log_info("[nex] hotkey listener restarted successfully");
                            if let Ok(mut guard) = self.hotkey_listener.lock() {
                                *guard = Some(new_listener);
                            }
                        }
                        Err(e) => {
                            log_warn(&format!("[nex] hotkey listener restart failed: {e}"));
                        }
                    }
                }
            }
        }
    }
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| search worker thread → Tantivy FFI | Unrecoverable panics from Tantivy C internals cross into Rust via FFI |
| filesystem → icon decode | Malformed/corrupt image files read from disk enter the `image` crate decoder |
| hotkey thread → Windows kernel | `GetMessageW` / `RegisterHotKey` can fail due to system state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Denial of Service | search_worker.rs::search thread | mitigate | `catch_unwind` around Tantivy search call; convert panics to error `SearchResult` instead of killing thread |
| T-06-02 | Denial of Service | icons.rs::decode() | mitigate | `catch_unwind` around `image::load_from_memory`; return None fallback (placeholder icon) |
| T-06-03 | Denial of Service | overlay/hotkey.rs::run_get_message_loop | mitigate | Thread exit detection via `is_alive()`; auto-restart with exponential backoff logging |
| T-06-04 | Information Disclosure | search_worker.rs panic payload | mitigate | Extract only `&str`/`String` message from panic; do not leak full backtrace in error results |
</threat_model>
