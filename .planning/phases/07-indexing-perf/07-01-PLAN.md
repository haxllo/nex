<objective>
Replace full-index rebuilds with incremental sync in both Tantivy and FTS5 search backends. The current `sync_indexes_from_cache()` calls `index_items()` which does a costly DELETE ALL + reindex of every item on every sync. This causes unnecessary I/O and CPU churn, especially when only a handful of items changed.

Purpose: Dramatically reduce index sync latency from O(n) full rebuild to O(delta) incremental update. The infrastructure already exists — `TantivyIndex.upsert_item()` and `delete_item()` are used by the file watcher consumer. This plan bridges the gap so the initial cache→index sync also uses incremental operations.

Output: Incremental sync methods on both TantivyIndex and Fts5Index, wired into CoreService.
</objective>

<task type="auto">
  <name>task 1: Add incremental_sync_items to TantivyIndex</name>
  <files>apps/core/src/tantivy_search.rs</files>
  <action>
Add a new public method `incremental_sync_items(&self, items: &[SearchItem]) -> Result<(), String>` to `TantivyIndex` that performs an incremental sync instead of the full `delete_all + rebuild` in `index_items()`.

**Algorithm:**
1. Lock the writer mutex (`self.writer.lock()`)
2. Collect all existing document IDs from the index: iterate the searcher's documents, extract the `id` field from each into a `HashSet<String>`
3. Collect all incoming item IDs into another `HashSet<String>`
4. **Delete removed items**: For each existing ID NOT in the incoming set, call `writer.delete_term(Term::from_field_text(self.fields.id, &existing_id))`
5. **Add/update items**: For each incoming item, call `writer.delete_term(Term::from_field_text(self.fields.id, &item.id))` followed by `writer.add_document(doc!(...))` (same doc! macro as in `index_items`)
6. `writer.commit()`
7. `let _ = writer.garbage_collect_files().wait();`

**Why this approach (not per-item commit):** A single commit for the entire batch avoids excessive segment creation. Per-item commits (like the current `upsert_item()`) create one segment per call, which bloats the index. The file watcher already uses per-item commits — that's fine for low-frequency real-time updates but should be fixed in Plan 07-02 (compaction).

**Edge cases:**
- Empty index (no existing docs): just add all items
- Empty items list: just delete all (equivalent to clear)
- `num_docs()` must return correct count after sync

**Keep `index_items()` unchanged** — it's used by tests and should remain available as a full-rebuild escape hatch.
</action>

Add a new public method `incremental_sync_items(&self, items: &[SearchItem]) -> Result<(), String>` to `Fts5Index` that performs an incremental sync instead of `clear() + rebuild` in `index_items()`.

**Algorithm:**
1. Collect all incoming item IDs into a `HashSet<String>`
2. Query existing IDs: `SELECT id FROM item_fts5` → collect into another `HashSet<String>`
3. **Delete removed items**: For each existing ID NOT in the incoming set, `DELETE FROM item_fts5 WHERE id = ?`
4. **Add/update items**: For each incoming item, execute `INSERT OR REPLACE INTO item_fts5 (id, title, path, subtitle, kind, extension) VALUES (?1, ?2, ?3, ?4, ?5, ?6)`
5. Use a single transaction: wrap steps 3-4 in `BEGIN IMMEDIATE` / `COMMIT` via `conn.execute_batch("BEGIN IMMEDIATE; ...; COMMIT;")` or use `conn.execute` within a transaction. The key is that deletes + inserts happen atomically.

**Edge cases:**
- Empty FTS5 table: just insert all items
- Empty items list: `DELETE FROM item_fts5` (equivalent to clear)
- `num_docs()` must return correct count after sync

**Keep `index_items()` unchanged** — used by tests and as full-rebuild escape hatch.
</action>

Modify `CoreService::sync_indexes_from_cache()` (line ~998) to call the new `incremental_sync_items()` methods instead of `index_items()`.

**CoreService change:**
In `sync_indexes_from_cache()`, replace:
```rust
if let Some(ref idx) = *tantivy_guard {
    if let Err(e) = idx.index_items(&items) { ... }  // FULL REBUILD
}
```
with:
```rust
if let Some(ref idx) = *tantivy_guard {
    if let Err(e) = idx.incremental_sync_items(&items) { ... }  // INCREMENTAL
}
```

Same pattern for FTS5: replace `idx.index_items(&items)` with `idx.incremental_sync_items(&items)`.

**IMPORTANT — first-sync guard:** On the very first sync after Tantivy/FTS5 index creation (when both were just opened fresh and have zero docs), `incremental_sync_items` with a large items list is equivalent to a full rebuild but without the efficiency of `delete_all_documents()` (which is faster than individual deletes for a massive clear). Track whether this is the first sync. `num_docs()` returning 0 on both backends means it's a fresh index — in that case, fall through to `index_items()` for the efficient bulk load. On subsequent syncs, use incremental.

Implementation: Before the Tantivy block, call `let is_first_sync = tantivy_guard.as_ref().map_or(true, |idx| idx.num_docs().unwrap_or(0) == 0);`. If `is_first_sync`, use `index_items()` (fast bulk load). Otherwise, use `incremental_sync_items()`.

**Discovery walk optimization (discovery.rs):**
In `discover_filesystem_walk()` (line ~517), the walk already breaks when `total_added >= total_budget || root_added >= per_root_budget`. However, `WalkDir::new(root).max_depth(max_depth).into_iter().filter_entry(...)` still traverses the directory tree to apply `filter_entry` — it just doesn't collect items past the limit. The optimization: after the limit is reached, break out of the outer `for entry in walkdir` loop entirely (already done via `break;` at line 559-561), AND stop the WalkDir iterator from further traversal. Since `WalkDir` uses a lazy iterator, the `break` already stops traversal. Verify this is working correctly — no code change needed if the break is effective; add a test that confirms walk stops early.

**DO NOT add any "first sync bulk load" logging beyond the existing `[nex] sync_indexes` log line. DO NOT change the `index_items()` methods — they remain available as the bulk-load path for first sync.**
