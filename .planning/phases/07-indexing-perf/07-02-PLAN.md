<objective>
Replace the current per-item commit strategy in Tantivy with batched commits and add index compaction (merge policy + periodic optimize). The current code commits after every single upsert/delete, creating excessive index segments and disk churn. This plan introduces: (1) batch commit counter with threshold, (2) Tantivy `LogMergePolicy` tuning, (3) periodic FTS5 `OPTIMIZE`, and (4) a compaction trigger wired into the sync flow.

Purpose: Reduce index fragmentation, reclaim disk space from deleted documents, and lower steady-state I/O overhead. This directly supports R3.2 (memory footprint) by preventing index bloat and R3.1 (indexing performance) by reducing segment merge storms.

Output: Compact, merge-policy-controlled Tantivy index + periodically optimized FTS5 index.
</objective>

  <action>
Modify `TantivyIndex` to use batched commits instead of committing after every individual write operation.

**Step 1 — Add write counter to TantivyIndex struct:**
```rust
pub(crate) struct TantivyIndex {
    index_path: PathBuf,
    reader: IndexReader,
    writer: Mutex<IndexWriter>,
    fields: TantivyFields,
    schema: Schema,
    write_count: Mutex<u32>,          // NEW: tracks uncommitted writes
    commit_threshold: u32,            // NEW: commit after this many writes
}
```

Set defaults: `commit_threshold = 500` (from CONTEXT: "After every 500 writes or every 5 minutes, commit the Tantivy writer").

**Step 2 — Configure LogMergePolicy at open time:**
In `TantivyIndex::open()`, after creating the writer, call `writer.set_merge_policy(Box::new(merge_policy))` where `merge_policy` is a `LogMergePolicy` with:
- `set_max_merge_segment_size(50_000_000)` — 50MB max segment
- `set_level_log_merge(true)` — tiered merging
- `set_min_num_segments_in_merge(2)` — merge at least 2 segments
- `set_max_num_segments_in_merge(8)` — no more than 8 at once

The `tantivy::merge_policy::LogMergePolicy` is imported from `tantivy::merge_policy`.

**Step 3 — Defer commits in upsert_item and delete_item:**
- Remove the explicit `writer.commit()` calls from `upsert_item()` and `delete_item()`
- After each write, increment `self.write_count` (lock the Mutex)
- After incrementing, if `write_count >= commit_threshold`, call `writer.commit()`, `writer.garbage_collect_files().wait()`, and reset `write_count` to 0
- Do NOT call `garbage_collect_files()` on every commit — only when write_count resets (to batch the GC cost)

**Step 4 — Add explicit flush method:**
```rust
pub fn flush(&self) -> Result<(), String> {
    let mut writer = self.writer.lock().map_err(|e| format!("..."))?;
    writer.commit().map_err(|e| format!("..."))?;
    let _ = writer.garbage_collect_files().wait();
    let mut count = self.write_count.lock().map_err(|e| format!("..."))?;
    *count = 0;
    Ok(())
}
```

**Step 5 — Call flush in the right places:**
- `incremental_sync_items()` already commits at the end (from Plan 07-01). Replace that explicit `writer.commit()` with `self.flush()` (or keep as is — either way, a full sync flushes).
- `index_items()` still needs its final commit — it already calls `writer.commit()` + `garbage_collect_files()` at the end of the batch. Leave that as-is (it's the bulk path).

**Step 6 — Update `incremental_sync_items()` (from 07-01):**
- The method was added in 07-01. It must call `flush()` (or commit directly) at the end so all writes are visible.
- After `flush()`, reset the write counter.

**IMPORTANT — Search visibility:** With deferred commits, newly indexed documents won't be visible to search until a commit happens. The `reader.reload()` call in `search()` only sees committed segments. This is acceptable because:
- File watcher updates are eventually consistent (sub-second delay is fine)
- `sync_indexes_from_cache()` explicitly flushes after the sync batch
- The search worker already calls `reader.reload()` before each search

**Keep all existing tests passing.** The deferred-commit behavior means tests that call `upsert_item()` then immediately `search()` need to call `flush()` between them (or the test must account for eventual consistency). Update existing unit tests in `mod tests` to call `flush()` where needed.

  <files>apps/core/src/fts5_search.rs, apps/core/src/core_service.rs</files>

**Part A — FTS5 optimize method:**

Add `pub fn optimize(&self) -> Result<(), String>` to `Fts5Index`:
```rust
pub fn optimize(&self) -> Result<(), String> {
    self.conn
        .execute("INSERT INTO item_fts5(item_fts5) VALUES('optimize')", [])
        .map_err(|e| format!("FTS5 optimize error: {e}"))?;
    Ok(())
}
```

FTS5 `optimize` reorganizes the index to reclaim space from deleted documents and merge B-trees. Unlike Tantivy, FTS5 doesn't need per-write commit deferral — SQLite handles that internally. We just need periodic `OPTIMIZE` calls.

**Part B — Wire compaction trigger in CoreService:**

Add a `compaction_write_count: Mutex<u32>` field to `CoreService`. Increment it in `index_item_on_backends()` and `remove_item_from_backends()` after each backend write. When it reaches 500: call `maybe_compact_backends()`, which:
1. Calls `tantivy_index.flush()` (triggers commit + GC if writes are pending)
2. Calls `fts5_index.optimize()`
3. Resets the counter to 0

Add a `pub(crate) fn maybe_compact_backends(&self)` method that performs the above. This is called from:
- `sync_indexes_from_cache()` — after the incremental sync completes (flush any remaining writes from the sync batch itself)
- The counter-based trigger in `index_item_on_backends`/`remove_item_from_backends`

The compaction counter must be independent of TantivyIndex's internal `write_count` — the CoreService counter tracks *total* writes across both backends for the purpose of triggering periodic FTS5 optimize.

**Part C — Periodic time-based compaction:**

Add `last_compaction_time: Mutex<Option<Instant>>` to CoreService. In `maybe_compact_backends()`, check if 5 minutes have elapsed since last compaction (from CONTEXT: "every 5 minutes"). If so, run compaction regardless of the write counter. Update the timestamp.

Call `maybe_compact_backends()` from the existing periodic path — add it to `prune_stale_items_if_due()` (which already runs on a timer) so compaction happens periodically even when no writes are happening.

**DO NOT change the file watcher consumer** — it already batches its writes. The `flush_pending()` function collects all added/removed items and calls `upsert_item`/`delete_item_by_id` in batch. With the deferred-commit Tantivy changes from Task 1, these per-item calls will accumulate until the threshold triggers a commit automatically.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-03 | Denial of Service | Tantivy merge storm | mitigate | `LogMergePolicy` with max 50MB segment size prevents runaway merges. Max 8 segments per merge caps I/O spike. |
| T-07-04 | Data Loss | Deferred Tantivy commits | accept | Up to 500 writes or 5 minutes of data could be lost on crash. All data is reconstructable from discovery providers on next restart. No user data or configuration is stored in the search index. |
