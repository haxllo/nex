```rust
// Scoring tier constants:
const SCORE_EXACT: i64 = 30_000;
const SCORE_PREFIX: i64 = 24_000;
const SCORE_SUBSTRING: i64 = 18_000;
const SCORE_FUZZY: i64 = 12_000;

fn score_text(normalized_title: &str, query: &str) -> Option<TextScore> {
    if normalized_title.is_empty() || query.is_empty() { return None; }

    let length_penalty = (normalized_title.len() as i64 - query.len() as i64).abs();
    let compact_bonus = (query.len() as i64) * 45;

    if normalized_title == query {
        return Some(TextScore { score: SCORE_EXACT + compact_bonus - length_penalty, kind: Exact });
    }
    if normalized_title.starts_with(query) {
        return Some(TextScore { score: SCORE_PREFIX + compact_bonus - length_penalty, kind: Prefix });
    }
    if let Some(position) = normalized_title.find(query) {
        let position_penalty = (position as i64) * 3;
        return Some(TextScore { score: SCORE_SUBSTRING + compact_bonus - position_penalty - length_penalty, kind: Substring });
    }
    // FALLS THROUGH TO FUZZY:
    let (start_penalty, gap_penalty) = subsequence_penalties(normalized_title, query)?;
    Some(TextScore { score: SCORE_FUZZY + compact_bonus - gap_penalty * 8 - start_penalty - length_penalty, kind: Fuzzy })
}

fn subsequence_penalties(haystack: &str, needle: &str) -> Option<(i64, i64)> {
    // Returns (start_penalty, gap_penalty)
    // For each char in needle: finds it in haystack starting from last match position
    // start_penalty = position of first match (earlier = better)
    // gap_penalty = sum of gaps between consecutive matches (adjacent chars = 0 gap)
    // Returns None if any needle char is not found in haystack
    // Example: "notpad" in "notepad": n(0), o(1), t(2), p(4)gap=1, a(5)gap=0, d(6)gap=0 → start=0, gap=1
    // Example: "ntepd" in "notepad": n(0), t(2)gap=1, e(3)gap=0, p(4)gap=0, d(6)gap=1 → start=0, gap=2
}
```

**Current typo tolerance behavior:**
- Dropped character (e.g., "notpad" → "Notepad"): 1 char missing → gap penalty of 1 (one jump of 1) → fuzzy score = 12000 + compact_bonus - 8 - start - length
- Extra character (e.g., "notepadd" → "Notepad"): needle longer than haystack → `subsequence_penalties` fails to find the extra 'd' → returns None → item excluded
- Swapped characters (e.g., "ontepad" → "Notepad"): needle[0]='o', found at pos 1 (gap from start=1). Then 'n' at pos 0, but `next_start` is now 2 → 'n' NOT found beyond pos 2 → returns None → item excluded

**The gap for extra characters:** When query has extra chars, `subsequence_penalties` fails because the extra char doesn't exist in the haystack. Currently the item is simply excluded. This is the PRIMARY gap to fix.

**Part A — Audit the current typo tolerance:**

The current `subsequence_penalties()` only handles **dropped characters** (needle shorter than haystack). It does NOT handle **extra characters** (needle longer than haystack). When the user types an extra character in their query, the function returns `None`, and the item is excluded from results entirely.

**Part B — Fix: Extend subsequence_penalties for extra characters:**

Modify `subsequence_penalties()` to handle the case where the needle has more characters than the haystack. The key insight: we don't need to match every needle character — we can allow "extra" characters in the needle (typos where the user typed too many chars).

**Algorithm change in `subsequence_penalties()`:**

Currently the function returns `None` when any needle character can't be found. Change it to allow up to 2 unmatched needle characters (for queries ≥ 3 chars) by adding a "missed" counter:

```rust
fn subsequence_penalties(haystack: &str, needle: &str) -> Option<(i64, i64)> {
    let mut next_start = 0;
    let mut start_penalty: Option<i64> = None;
    let mut previous_position: Option<usize> = None;
    let mut gap_penalty = 0_i64;
    let mut missed_chars = 0_u32;
    let needle_len = needle.chars().count();
    
    // Allow up to 2 missed characters for queries of 3+ chars
    let max_missed = if needle_len >= 3 { 2 } else { 0 };

    for needle_char in needle.chars() {
        let mut found: Option<(usize, usize)> = None;
        for (offset, hay_char) in haystack[next_start..].char_indices() {
            if hay_char == needle_char {
                let absolute = next_start + offset;
                found = Some((absolute, hay_char.len_utf8()));
                break;
            }
        }

        match found {
            Some((position, char_len)) => {
                if start_penalty.is_none() {
                    start_penalty = Some(position as i64);
                }
                if let Some(previous) = previous_position {
                    gap_penalty += position.saturating_sub(previous + 1) as i64;
                }
                previous_position = Some(position);
                next_start = position + char_len;
            }
            None => {
                // This needle char not found — count as missed
                missed_chars += 1;
                if missed_chars > max_missed {
                    return None; // Too many unmatched chars
                }
                // Apply a penalty for each missed character
                gap_penalty += 3; // penalty equivalent to skipping ~3 positions
                // Don't advance next_start — the next char may still match nearby
            }
        }
    }

    Some((start_penalty.unwrap_or(0), gap_penalty))
}
```

**Why this works:**
- Query "notepadd" (9 chars) against "notepad" (7 chars normalized): n-o-t-e-p-a matched (gap=0), first 'd' matched, second 'd' → missed_chars=1 (≤2) → gap_penalty += 3
- Result: fuzzy score = 12000 + 9*45 - 3*8 - start_penalty - length_penalty = 12000 + 405 - 24 = 12381
- Query "firefoxx" against "firefox": all chars match, extra 'x' → missed_chars=1 → +3 gap penalty → still scores well
- Query "xy" against something: needle_len=2, max_missed=0 → no tolerance (existing behavior, correct per R3.3: typo tolerance only for ≥3 chars)

**Part C — Keep backward compatibility:**

The `max_missed` threshold prevents the change from affecting short queries (1-2 chars). The existing behavior for dropped-character queries is unchanged. The only new behavior is: queries with 1-2 extra characters now succeed (with a modest gap penalty) instead of failing entirely.
