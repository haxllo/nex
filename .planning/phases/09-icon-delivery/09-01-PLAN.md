## Problem

`snapshot_json()` in `host.rs` base64-encodes all row icons inline in the JSON payload while holding `state.lock()`. For 20 rows this produces ~134KB of JSON and holds the mutex for ~2-5ms of base64 encoding + serialization — blocking the runtime worker from updating rows or selection during that window.

Current flow (under lock):
```
lock(state) → iterate rows → base64 encode each icon → build JSON → drop lock → PostWebMessageAsJson
```

## Solution

Split `snapshot_json()` into two phases: a lightweight state message (no icons) and a separate icon data message. Both use `PostWebMessageAsJson` (fire-and-forget, non-blocking). The state lock is released before any icon encoding occurs.

New flow:
```
lock(state) → clone state → drop lock → build lightweight JSON (~2KB) → PostWebMessageAsJson #1
                                       → base64 encode icons outside lock → PostWebMessageAsJson #2
```

The JS side receives both messages in rapid succession (same frame). It maintains a `Map<path, dataUri>` icon cache that persists across renders, so repeated pushes with the same icons skip re-encoding and re-rendering `<img>` elements.

**Note on blocking decode:** `IconCache::png_bytes()` (the blocking version) is used for icon encoding, not `png_bytes_cached()`. This means the first few icons of a cold session will block the encoding thread while decoding from disk/COM. This is acceptable because the blocking now happens **outside the state lock** — the runtime worker is free to process events during icon decoding. The existing `prefetch_rows()` background thread mitigates this for the common case (icons decoded before first push).

#### a. Extract `post_json()` helper

#### b. Refactor `push_state()` — split into two phases

Current function:
```rust
fn push_state(webview: &Option<WebView>, state: &Arc<Mutex<ShimState>>, icons: &Arc<IconCache>) {
    let Some(wv) = webview else { return };
    let Ok(s) = state.lock() else { return };
    let json = snapshot_json(&s, icons);  // ← heavy work under lock
    drop(s);
    // PostWebMessageAsJson

```

New function — clone state under lock, serialize outside:
```rust
fn push_state(webview: &Option<WebView>, state: &Arc<Mutex<ShimState>>, icons: &Arc<IconCache>) {
    let Some(wv) = webview else { return };

    // Phase 1: Clone state under lock (microseconds).
    let snapshot = {
        let Ok(s) = state.lock() else { return };
        s.clone()
    };

    // Phase 2: Build lightweight JSON without icons (~2KB).
    let state_json = snapshot_state_json(&snapshot);

    // Phase 3: Encode icons outside lock (~2-5ms for 20 rows).
    // Note: png_bytes() may block on first decode per icon (cold cache),
    // but the state lock is not held during this work.
    let icons_json = snapshot_icons_json(&snapshot, icons);

    // Phase 4: Send both messages back-to-back (same frame).
    post_json(&wv, &state_json);
    if !icons_json.is_empty() {
        post_json(&wv, &icons_json);
    }
}
```

#### d. New `snapshot_state_json()` — lightweight, no icons

Returns the full state payload with icon fields set to the icon path string (used as the JS cache key). Null if no icon:

```rust
fn snapshot_state_json(s: &ShimState) -> String {
    let rows: Vec<serde_json::Value> = s.rows.iter().map(|r| {
        let role = match r.role {
            OverlayRowRole::Header => "header",
            OverlayRowRole::Status => "status",
            OverlayRowRole::Calculator => "calculator",
            OverlayRowRole::TopHit | OverlayRowRole::Item => "item",
        };
        let selectable = matches!(
            r.role,
            OverlayRowRole::Item | OverlayRowRole::TopHit | OverlayRowRole::Calculator
        );
        serde_json::json!({
            "role": role,
            "title": r.title,
            "subtitle": r.path,
            "kind": r.kind,
            "icon": if r.icon_path.is_empty() { serde_json::Value::Null } else { serde_json::Value::String(r.icon_path.clone()) },
            "selectable": selectable,
            "resultIndex": r.result_index,
        })
    }).collect();

    let theme = match s.theme {
        Theme::Dark => "dark",
        Theme::Light => "light",
    };

    serde_json::json!({
        "query": s.query,
        "rows": rows,
        "selected": s.selected,
        "status": s.status_text,
        "placeholder": s.placeholder_hint,
        "hotkeyHint": s.hotkey_hint,
        "hotkeyIssue": s.hotkey_issue_active,
        "theme": theme,
    }).to_string()
}
```

#### e. New `snapshot_icons_json()` — icon data only (with dedup)

Returns a JSON object mapping icon paths to data URIs. Empty string if no icons.
Deduplicates by path to avoid encoding the same icon twice when multiple rows share a path.

```rust
fn snapshot_icons_json(s: &ShimState, icons: &Arc<IconCache>) -> String {
    let mut seen = std::collections::HashSet::new();
    let icon_map: serde_json::Map<String, serde_json::Value> = s.rows.iter()
        .filter(|r| !r.icon_path.is_empty())
        .filter(|r| seen.insert(r.icon_path.clone()))  // dedup shared paths
        .filter_map(|r| {
            let b64 = icons.png_bytes(&r.icon_path)
                .map(|arc| base64_data_uri(arc.as_ref()))
                .unwrap_or_default();
            if b64.is_empty() { None }
            else { Some((r.icon_path.clone(), serde_json::Value::String(b64))) }
        })
        .collect();

    if icon_map.is_empty() {
        return String::new();
    }

    serde_json::json!({ "icons": icon_map }).to_string()
}
```

#### f. Remove old `snapshot_json()` function

The monolithic function is replaced by `snapshot_state_json()` + `snapshot_icons_json()`.

#### c. New `patchIcons()` function — NO placeholder guard

Iterates existing `<img>` elements and updates `src` from the cache.
Does **not** skip placeholder elements — on cold cache, `render()` creates icons
without setting `src`, and `patchIcons()` must update them all.

```javascript
function patchIcons() {
    for (const li of list.children) {
        const img = li.querySelector("img.icon");
        if (!img) continue;
        const path = img.dataset.iconPath;
        if (path && iconCache.has(path)) {
            const dataUri = iconCache.get(path);
            if (img.src !== dataUri) img.src = dataUri;
            img.classList.remove("placeholder");
        }
    }
}
```

#### d. Update `render()` to use icon cache — no premature placeholder

In the row rendering loop, change icon handling. Do **not** add `placeholder` class
for icons that have a path but aren't cached yet — `patchIcons()` will set the src
and the browser handles the loading state. Only `onerror` should trigger placeholder.

**Current:**
```javascript
if (r.icon && r.kind !== "action") {
    const img = document.createElement("img");
    img.className = "icon";
    img.src = r.icon;  // ← r.icon was a data URI, now it's a path
    img.onerror = () => img.classList.add("placeholder");
    li.appendChild(img);
}
```

**New:**
```javascript
if (r.icon && r.kind !== "action") {
    const img = document.createElement("img");
    img.className = "icon";
    img.dataset.iconPath = r.icon;  // store path for patchIcons()
    if (iconCache.has(r.icon)) {
        img.src = iconCache.get(r.icon);
    }
    // Don't add placeholder class here — patchIcons() will set src
    // and the browser handles loading. Only onerror triggers placeholder.
    img.onerror = () => img.classList.add("placeholder");
    li.appendChild(img);
} else if (r.kind !== "action") {
    const ph = document.createElement("div");
    ph.className = "icon placeholder";
    li.appendChild(ph);
}
```

## Message Sequence Diagram

```
Rust (host.rs)                          JS (app.js)
     │                                       │
     ├── lock(state)                          │
     ├── clone state (ShimState)              │
     ├── unlock(state)                        │
     │                                       │
     ├── build state JSON (~2KB)              │
     ├── PostWebMessageAsJson(state)  ───────>│── apply(state)
     │                                       │   ├─ render rows (icons without src)
     │                                       │
     ├── base64 encode icons (~2-5ms)         │
     │   (outside lock — worker free)         │
     ├── build icons JSON (~134KB)            │
     ├── PostWebMessageAsJson(icons) ────────>│── apply(icons)
     │                                       │   ├─ merge into iconCache
     │                                       │   └─ patchIcons() update <img> src
     │                                       │       + remove placeholder class
```

Both messages arrive in the same frame (back-to-back `PostWebMessageAsJson`).
The user sees rows with icons on the first paint — `patchIcons()` runs before
the next vsync.

## Performance Impact

| Metric | Before | After |
|--------|--------|-------|
| State lock hold time | ~2-5ms (includes icon encoding) | ~0.1ms (clone only) |
| First message size | ~134KB (full payload) | ~2KB (no icons) |
| Second message size | — | ~134KB (icons only) |
| Total bytes sent | ~134KB | ~136KB (negligible difference) |
| Duplicate icon encoding | N/A (no dedup) | Eliminated (HashSet dedup) |
| JS icon cache hit rate | 0% (re-decodes every push) | ~80%+ (same icons reused) |
| DOM rebuild for icons | Full (every push) | Patch only (existing elements) |

The primary win is **lock contention reduction** (2-5ms → ~0.1ms). The secondary win is **JS-side icon cache** preventing unnecessary `<img>` src reassignment on repeated pushes.

## Risks and Mitigations

| Risk | Mitigation |
|------|------------|
| Two messages processed in different frames (icon flash) | PostWebMessageAsJson is sequential IPC — both delivered in same task batch |
| JS cache grows unbounded | Cache entries match active rows only; clear on hide (teardown destroys WebView) |
| `clone()` of ShimState is expensive | ShimState is small (~200 bytes); clone is ~microseconds, negligible vs encoding |
| Icon path as string key has encoding issues | Paths are filesystem-native strings, no URL encoding needed |
| Blocking `png_bytes()` on cold cache | State lock is not held; prefetch thread mitigates for common case |
| Shared icon paths encoded twice | `HashSet` dedup in `snapshot_icons_json` prevents redundant work |

## Future Optimizations (not in scope)

- **Hash-based deduplication:** Send icon content hashes in state push; JS replies with missing hashes; Rust sends only new icons. Reduces second message to ~0 after cache warm.
- **Deferred icon delivery:** When prefetch completes after initial push, send only newly-available icons via a separate message path.
- **Icon prefetch cancellation:** Use a single background thread with work queue instead of spawning a new thread per `set_results()` call.
